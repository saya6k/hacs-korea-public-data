# `fuel` (유가정보 - Opinet)

Secret field is `opinet_api_key` (renamed from `api_key` in 4.9 — `__init__.py` and the options flow always try `opinet_api_key` first, then fall back to `api_key` for pre-4.9 entries; don't remove that fallback).

**Regions are config subentries, unlike `pharmacy`.** Each 시도 is a `fuel_region` subentry (`{"sido_code", "fuel_codes"}` — `fuel_codes` is a multi-select, so one region can track several fuel types at once), added via the initial multi-sido `fuel` step (same `fuel_codes` selection applied to every sido picked) or the "지역 추가" `FuelRegionSubentryFlowHandler` (one sido + its own `fuel_codes` at a time). Re-adding an already-configured sido aborts with `already_configured` — to change an existing region's fuel types, use its subentry's **reconfigure** action (`FuelRegionSubentryFlowHandler.async_step_reconfigure`), not the add flow; there was a bug where users tried "지역 추가" again on an existing sido to add a fuel type and it silently did nothing (aborted) — reconfigure is the fix.

There's a single `FuelCoordinator` per entry, not one per subentry — it fetches the nationwide average once plus one low-price call per (sido, fuel) pair regardless of which subentry contributed it, so splitting by subentry would only duplicate the average fetch. `__init__.py` flattens `entry.subentries` into the same `configs: [{"sido_code", "fuel_code"}, ...]` shape the coordinator always expected; `sensor.py` re-groups by subentry for `config_subentry_id`. Legacy flat entries (`entry.data["configs"]`, or the oldest single `sido_code`/`fuel_code` pair, from before the 4.9 subentry refactor) still load with no subentries — don't remove those fallbacks.

Each (sido, fuel) pair gets three sensors: `FuelAvgSensor`, `FuelLowSensor`, and `FuelLowLocationSensor` (state = lowest-price station name, `latitude`/`longitude` attributes for map cards — converted from Opinet's KATEC `GIS_X_COOR`/`GIS_Y_COOR` via `fuel/coords.py`'s pure-Python inverse-TM + Bursa-Wolf datum shift; Opinet does not return WGS84 directly, unlike `pharmacy`'s `wgs84Lat`/`wgs84Lon`). The KATEC conversion was verified against `pyproj` at 3 reference points (Seoul/Busan/Jeju, sub-cm error) during development — if you touch `fuel/coords.py`, re-verify the same way rather than trusting the algebra by eye; an earlier draft had a ~30m systematic error from a Bursa-Wolf rotation-sign convention mistake.
