# `earthquake` (지진 정보)

The simplest service: nationwide 기상청 지진정보조회서비스 feed, no region concept at all — single entry, single `EarthquakeCoordinator`, no subentries. `event` platform only (`EarthquakeEvent` in `earthquake/sensor.py`, despite the filename).

Referenced elsewhere in this repo as the baseline for "does this service actually need subentries/regions, or can it just be one flat entry" — e.g. `pharmacy`'s 4.9 rewrite deliberately moved toward this level of simplicity instead of `disaster`'s region-subentry approach.

The entry does carry one location-like concept though: a home point + radius (km) + minimum magnitude, used by `EarthquakeEvent._handle_coordinator_update` to decide whether a fetched earthquake is close/strong enough to fire the `earthquake_alert` event (`earthquake/api.py:haversine_km`). Both the config flow (`async_step_earthquake`) and the options flow (`_build_schema`'s `ENTRY_EARTHQUAKE` branch) collect this via a single `LocationSelector(LocationSelectorConfig(radius=True))` field named `location` — a map pin + draggable radius circle — rather than raw lat/lon/radius number inputs. The selector's radius is in **meters**; it's divided by 1000 before being stored as `entry.data["radius_km"]` (and multiplied back by 1000 when re-showing the options form). The map's *initial* pin position defaults to `hass.config.latitude/longitude` for a brand-new entry, but once saved, `home_latitude`/`home_longitude` in `entry.data` are authoritative and don't track `zone.home` afterward. `KRPublicDataOptionsFlow.async_step_init` has a special-case that unpacks the submitted `location` dict into `home_latitude`/`home_longitude`/`radius_km` before the generic merge — `entry.data` itself never stores a `location` key.
