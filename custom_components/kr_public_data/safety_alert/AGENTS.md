# `safety_alert` (안전알림 - 행정안전부)

Regions live flatly in `entry.data["regions"]` (`[{"code", "name"}, ...]`) — no subentries, same style `pharmacy` moved to in 4.9. One `SafetyAlertCoordinator` per region, keyed by region `code` in `store["coordinators"]`. Legacy fallback: a pre-list single-region shape (`entry.data["area_code"]`/`["area_name"]`) — don't remove it.

Region names/codes come from the safekorea region API (`safety_alert/region_api.py`), the same lookup `disaster` and (pre-4.9) `pharmacy` use for their 시군구 pickers.

The region picker itself (시도 + 서울 자치구, `config_flow.py`'s `SAFETY_ALERT_REGIONS`) is a static dict, not a live API call — shared between the initial `async_step_safety_alert` creation step and the options flow's `ENTRY_SAFETY_ALERT` schema so they can't drift out of sync (they used to be two separate inline copies, and the options one silently dropped the Seoul-gu entries). Editing the options-flow `area_codes` multi-select rewrites `entry.data["regions"]` wholesale (`KRPublicDataOptionsFlow.async_step_init` converts the submitted codes to `{"code", "name"}` items before merging) — until this was wired up, editing that screen did nothing at all, since `__init__.py` only ever reads `regions`. Because a whole-list rewrite can drop a region without deleting anything (there's no subentry here to delete), `safety_alert/cleanup.py:async_cleanup_stale_safety_alert_entities()` runs at the end of every `ENTRY_SAFETY_ALERT` setup and removes any entity/device for a region code no longer in the list — same forward-diff approach as `weather/cleanup.py`.
