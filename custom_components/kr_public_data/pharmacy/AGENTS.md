# `pharmacy` (약국 운영정보)

Two actions are registered globally on entry setup (only one pharmacy entry should exist per HA instance): `search_pharmacy` (ad-hoc region search, calls the API live, bypasses the coordinator) and `list_nearby_pharmacies` (aggregates every region's already-fetched location-sensor pharmacies into one response — see below). Both live in `pharmacy/services.py`; `api_key` is editable via the options flow.

**Deliberately not subentry-based** (unlike `disaster`, which has a near-identical region shape): regions live flatly in `entry.data["regions"]` (`[{"sido", "sgg", "location_sensors", "location"}, ...]`), all created in one shot via the `pharmacy_sgg` checklist step in the config flow — there is no "add region" flow, so adding/removing a district means recreating the entry. Editing an *existing* region's `location_sensors`/`location` is supported, though, via the options flow (`KRPublicDataOptionsFlow.async_step_pharmacy_options`): single-region entries edit in place on the same screen as `api_key`; multi-region entries pick a region first (`async_step_pharmacy_options`'s `region` dropdown) then edit that one (`async_step_pharmacy_region_edit`) — the other regions are left untouched. `PharmacyCoordinator`s are one-per-region (the API queries per 시군구), keyed by list index in `store["coordinators"]`, not by subentry id.

`location` is a `LocationSelector(LocationSelectorConfig(radius=True))` — a map widget where the user drops a pin and drags a radius circle, submitted as `{"latitude", "longitude", "radius"}`. **It does not default to `zone.home` for existing regions** — only a brand-new region's map initially centers on `hass.config.latitude/longitude` as a starting point; once saved, `region["location"]` is authoritative and independent of wherever `zone.home` moves afterward. `pharmacies_within_radius()` in `pharmacy/sensor.py` (built on `homeassistant.util.location.distance`) is called with `region["location"]`'s values, not `hass.config` directly — check `sensor.py`'s `ENTRY_PHARMACY` branch for the exact fallback order below before assuming a `location_sensors` bug.

Three legacy fallbacks apply, oldest first, when the newer shape is absent — don't remove any of them:
1. Flat regions with the old scalar `radius` field but no `location` dict (from before `location` became a map-pin selector) — `sensor.py` falls back to `region.get("radius", 1000)` + `hass.config.latitude/longitude` for these, reproducing the old zone.home-relative behavior exactly.
2. Pre-4.9 subentry-based pharmacy entries (`entry.subentries`, from before the flat-entry rewrite) — each region carries a synthesized `subentry_id` so its entities still land under `config_subentry_id` and existing devices survive.
3. The oldest pre-subentry single-region shape (`entry.data["q0"]`/`["q1"]`).

Setting `location_sensors: true` on a region spawns, per pharmacy within the region's radius, one `PharmacyLocationSensor` (state = address, `latitude`/`longitude` attributes for map cards) **and** one `PharmacyOpenBinarySensor` (state = currently-open, derived from that pharmacy's `duty_time` via `pharmacy/binary_sensor.py:is_pharmacy_open()` — parses the `"HHMM~HHMM"` strings built in `api.py`, handles missing days and overnight/24h wraparound). This can add a lot of entities, which the config flow's `data_description` warns about — **but they're not scattered as one device each**: both entity types share the *same* `device_info` as that region's `PharmacySensor` (`pharmacy/device.py:pharmacy_region_device(sido, sgg)`), so opening the one "약국 - {지역}" device shows the count sensor plus every nearby pharmacy's location + open-now entities together, rather than one device per pharmacy. `list_nearby_pharmacies` is the complementary non-UI view: it walks every pharmacy entry's `location_sensors`-enabled regions fresh at call time (not a stale captured reference — it re-reads `hass.data[DOMAIN]` per entry so it survives entry reloads) and returns one combined `{"pharmacies": [...], "count": N}`, for automations/scripts that want the list without touching entities at all.

Opinet-adjacent note: unlike `fuel`, this API (`ErmctInsttInfoInqireService`) returns `wgs84Lat`/`wgs84Lon` directly — no KATEC coordinate conversion needed.

`LocationSelector` limitation (verified against `home-assistant/frontend` source, not just docs): it does **not** render existing zones (e.g. `zone.home`) as reference markers on the map — `ha-selector-location.ts` passes only the single value being edited into `ha-locations-editor`, no `hass.states` zone lookup at all. There's no selector config flag to enable this; it would require a custom frontend component, out of reach for a config-flow-only integration. Don't promise this to users without re-checking upstream first.
